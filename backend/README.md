# PromApparat Workspace API

Серверный контур общей рабочей системы компании.

## Что хранится на сервере

- пользователи и роли `director` / `manager`;
- общее состояние рабочего пространства в PostgreSQL;
- ревизии и защита от одновременной перезаписи;
- журнал серверных изменений;
- метаданные и содержимое загруженных файлов;
- refresh-токены пользовательских сессий.

## Локальный запуск

Из корня репозитория:

```bash
docker compose up --build
```

После запуска:

- frontend: `http://localhost:5173`
- API: `http://localhost:8787`
- PostgreSQL: `localhost:5432`

Локальные тестовые учётные записи задаются в `docker-compose.yml`:

- директор: `director@promapparat.local`
- менеджер: `manager@promapparat.local`

Пароли из compose предназначены только для локальной разработки и должны быть заменены перед любым внешним размещением.

## Ручной запуск backend

```bash
cd backend
cp .env.example .env
npm install
npm run migrate
npm run seed
npm start
```

## Переменные окружения production

Обязательные:

- `DATABASE_URL` — PostgreSQL connection string;
- `DATABASE_SSL=true` для управляемой облачной базы, если требуется;
- `JWT_SECRET` — случайная строка не короче 32 символов;
- `CORS_ORIGINS` — адрес frontend без завершающего слэша;
- `FILE_STORAGE_DIR` — путь к постоянному диску;
- `SEED_ON_START=false` после создания первых пользователей.

Для первого запуска можно временно включить `SEED_ON_START=true` и задать `SEED_DIRECTOR_*`, затем немедленно отключить автоматический seed и сменить пароль директора через API/интерфейс.

## Хранение файлов

Файлы записываются в `FILE_STORAGE_DIR`, а метаданные — в PostgreSQL. В production директория должна находиться на постоянном диске/volume. Эфемерный файловый диск приведёт к потере PDF и Excel после перезапуска контейнера.

Следующий инфраструктурный шаг — заменить файловый адаптер на S3-совместимое хранилище. Ссылки в рабочем пространстве уже используют серверные идентификаторы файлов, поэтому frontend переделывать не потребуется.

## Модель синхронизации

Frontend отправляет `baseRevision` и новое состояние. Сервер блокирует строку workspace, сравнивает ревизию и:

- сохраняет разрешённые для роли разделы;
- увеличивает ревизию;
- записывает изменённые разделы в `audit_log`;
- возвращает `409 REVISION_CONFLICT`, если другой сотрудник сохранил изменения раньше.

Менеджер не может менять финансовые настройки и формулы. Директор не может через workspace-sync изменять операционные сделки, позиции, поставки и оплаты.
